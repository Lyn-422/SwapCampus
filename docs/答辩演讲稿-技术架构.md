# 技术架构 — 答辩演讲稿

> 讲解时长：约 3 分钟 | 配合架构图讲述

---

各位老师好，接下来我花三分钟讲一下 SwapCampus 的技术架构。我会用大白话解释这张图，保证在座都能听懂。

---

## 开场：一句话概括

**这张图说的就是——用户打开浏览器，点了几下，背后发生了什么。**

整个系统分三层：左边是前端（用户看到的界面），中间是后端（干活的），下边是存储层（记数据的）。它们之间通过两条线连着：一条叫 HTTP，负责普通请求；一条叫 WebSocket，负责聊天。

---

## 第一部分：前端 — 用户看到什么

先看左上角这块——Vue 3 前端。

在座都用过闲鱼或者转转对吧，就是那个感觉。用户打开 geleme.online，看到一个网页，上面有商品列表、搜索框、聊天窗口——这些界面全是用 Vue 3 写的。它叫 SPA，意思是"单页面应用"——你点来点去，页面不会刷新白屏，体验跟手机 App 一样顺滑。

配套用了三样东西：Element Plus 是 UI 控件库，按钮长什么样、表格长什么样不用自己画；Pinia 管状态，比如你登录了，整个网站都知道"这个人已登录"；Axios 负责跟后端要数据。

---

## 第二部分：后端 — 谁在干活

前端的界面再好看，数据得有人给。中间这块就是 Django 后端，所有业务逻辑都在这里。

它分两个通道：
- **左边那条 HTTP 线**——你搜商品、下订单、登录注册，走这条路。用的是 Waitress，你可以理解为普通窗口，一个请求进来处理完再接下一个。
- **右边那条 WebSocket 线**——你聊天，走这条路。用的是 Daphne，它专门保持长连接，消息一来立刻推送，不用你刷新。底下靠 Redis 做消息广播——你发一条消息，Redis 立马喊一声"对方快来看"，对方屏幕上就弹出来了。

打个比方：HTTP 就像寄信，寄出去等着回；WebSocket 就像打电话，一直通着，随时说随时听到。

---

## 第三部分：存储层 — 数据放在哪

后端处理完请求，数据往哪存？看图最下边三块：

- **MySQL** 是账本。用户信息、商品信息、订单记录，全记在这里。为什么用它？因为数据不能丢，关机重启还得在。
- **Redis** 是公告栏。聊天消息广播、热门数据缓存，读写字条级别的速度。但它只记临时的事，重启就没了。
- **MinIO** 是仓库。商品图片、用户头像、聊天发的图——这些大文件不能塞数据库，专门放 MinIO 里。它兼容亚马逊 S3，但免费。

---

## 第四部分：Nginx — 门口的管家

左边中间这块 Nginx，是整个系统的"守门人"。

所有请求先到它这儿，它看一眼你要什么：
- 要页面？直接给你，不用惊动后端，快。
- 要数据？转发给 Django 后端处理。
- 要聊天？识别出 WebSocket，转发给 Daphne。
- 要图片？直接从 MinIO 取。

它还能挡攻击、限流量、加 HTTPS 加密锁。没有它，就像奶茶店没有门迎，100 个人同时涌进来谁也服务不了。

---

## 第五部分：安全 — 怎么保证不出事

最后快速提一下安全，六道防线：
1. 密码用哈希存，数据库泄露了也看不到原密码
2. JWT Token 管登录，2 小时过期，不是偷了就能一直用
3. 用户只能改自己的数据，改不了别人的
4. 图片上传限制 20MB，服务端验格式，防止病毒文件
5. 前端自动防跨站脚本攻击
6. 数据库用参数化查询，防 SQL 注入

---

## 结尾：一句话回归

**这张图的核心就是：前端长什么样、后端怎么干活、数据怎么存、Nginx 怎么挡在前面——四个角色分工明确，各干各的，合在一起就是一个完整的网站。**
